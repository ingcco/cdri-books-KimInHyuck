# Karpathy 4원칙 — 행동 메타 원칙

`.claude/rules/conventions.md`(코드 스타일)와 카테고리가 다른 **행동 원칙**. 작업의 "어떻게"보다 "무엇을, 언제, 얼마나"를 다룬다. 출처: `andrej-karpathy-skills` 레포의 단일 CLAUDE.md.

> conventions가 *문법*이라면 Karpathy는 *태도*. 둘은 충돌하지 않으며, Karpathy가 메타로 적용된 후 conventions가 코드 레벨에 적용된다.

## 원칙 1: Think Before Coding

코드를 쓰기 전 **가정을 명시**하고 **트레이드오프를 제시**한다. 모호한 부분이 남아 있으면 **질문**한다.

- ❌ 사용자가 "검색 기능 추가해줘"라고 하면 즉시 input + filter 코드를 작성
- ✅ 사용자가 "검색 기능 추가해줘"라고 하면 "제목/저자/출판사 중 어느 대상인가요? 실시간 검색인가요, 버튼 검색인가요? 추천: 제목 대상 + 500ms debounce (이유: ...)"라고 질문

**자기 점검**:

- [ ] 사용자의 요청에 모호한 어휘(`추가`, `개선`, `정리`)가 있으면 한 번에 하나씩 결정 트리 분기를 확인했는가
- [ ] 두 가지 이상의 구현 방식이 가능하면 트레이드오프(성능/유지보수/일관성)를 명시했는가
- [ ] 가정(예: "기존 컴포넌트를 그대로 쓸 수 있다고 가정")을 plan 문서나 응답에 기록했는가

## 원칙 2: Simplicity First

**요청 이상**의 변경을 자발적으로 하지 않는다. 단일 사용 로직은 인라인 유지. 발생할 수 없는 시나리오에 대한 에러 처리를 추가하지 않는다.

- ❌ "버튼 색 바꿔줘" 요청에 ColorContext + ThemeProvider 도입
- ✅ "버튼 색 바꿔줘" → 해당 컴포넌트의 tailwind 클래스만 교체. 다른 곳 색상은 손대지 않음
- ❌ 내부 함수 호출에 try/catch 추가(이미 상위에서 잡힘)
- ✅ 외부 API/사용자 입력 경계에서만 검증

**자기 점검**:

- [ ] 추상화/헬퍼 함수를 추가했다면 **3곳 이상**에서 같은 로직을 쓰는가
- [ ] 에러 처리를 추가했다면 그 에러가 **실제로 발생 가능한 시나리오**인가 (시스템 경계인가)
- [ ] feature flag, fallback, backwards-compat shim 없이 직접 변경할 수 있는가

> 충돌 회피: conventions.md "최소 추상화(3곳 미만 인라인)"과 동일 방향. Karpathy가 메타 원칙, conventions가 구체 임계값.

## 원칙 3: Surgical Changes

작업 범위 **인접 코드를 자발적으로 "개선"하지 않는다**. 작동 중인 코드는 리팩토링하지 않는다. 기존 스타일을 따른다.

- ❌ `useBookSearch.ts`에서 `handleToggleFavorite` 분리 요청 → 같은 파일의 `handleSort`도 함께 분리
- ✅ `handleToggleFavorite`만 분리. `handleSort`는 발견사항으로 보고
- ❌ 변경한 파일 옆 import 순서가 마음에 들어 정렬
- ✅ 변경한 줄만 수정, import는 손대지 않음

**자기 점검**:

- [ ] git diff가 사용자가 요청한 변경에 **국한**되는가
- [ ] 인접 코드의 컨벤션 위반·raw HTML·import 정리를 "함께" 했다면, 그것을 별 ticket으로 분리할 수 있는가
- [ ] 에이전트 위임 시 "plan에 없는 변경은 전부 발견사항 보고" 문구를 프롬프트에 포함했는가

> 연계: `.claude/rules/ai-defense.md` "에이전트 위임 시 스코프 엄수" 절.

## 원칙 4: Goal-Driven Execution

각 단계에 **검증 가능한 기준**을 붙인다. 형식: `[단계] → 검증: [확인 방법]`.

- ❌ "검색 API 연동" (검증 불명)
- ✅ "카카오 도서 검색 query 연동 → 검증: 검색어 입력 시 documents 렌더 + MSW mock 응답으로 테스트 통과"
- ❌ "버그 수정" (재현 불가)
- ✅ "재현 테스트 작성 후 통과 → 검증: 해당 테스트가 fix 전 fail, fix 후 pass"

**자기 점검**:

- [ ] 각 plan Step이 `→ 검증:`을 가지는가
- [ ] 검증 방법이 **자동화 가능**(테스트/타입체크/grep)하거나 **명시적 수동 절차**(스크린샷 비교 등)인가
- [ ] "동작합니다"가 아닌 "X 명령어로 Y 결과 확인"으로 보고하는가

## 도서 검색 과제 적용 예시 3

| 상황                              | 잘못된 적용                                              | Karpathy 적용                                                                              |
| --------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| **버그 수정 요청**                | 증상만 고치고 다른 곳도 "혹시 모르니" 추가 방어 코드 삽입 | 재현 테스트 작성 → fail 확인 → fix → pass 확인. **검증 단계 명시** (원칙 4 + 원칙 3)        |
| **검색 페이지에 상세검색 추가**   | input + useState + filter 즉시 구현                      | "제목/저자/출판사 target 선택 여부, debounce 여부" 질문 → 결정 → plan 문서에 기록 (원칙 1)  |
| **찜 훅 옆 unused export 발견**   | 작업 김에 함께 제거하고 import 정리                      | 작업 범위 내 변경만 commit. unused export는 plan "발견 사항"에 backlog로 기록 (원칙 3)      |

## 다른 rule과의 관계

| 다른 rule            | 카테고리   | Karpathy와의 관계                                                            |
| -------------------- | ---------- | ---------------------------------------------------------------------------- |
| `conventions.md`     | 코드 스타일 | 메타 원칙(Karpathy) 적용 후 구체 임계값(3곳 미만 인라인, 파일명 규약 등) 적용 |
| `ai-defense.md`      | AI 방어     | "과도한 엔지니어링 방지", "에이전트 위임 스코프 엄수"는 Surgical Changes의 구체화 |
| `anti-patterns.md`   | 안티패턴    | Surgical Changes 위반(scope 초과 변경)·과도한 방어 코드를 카탈로그로 검출     |
