# 작업 플래닝 스킬

작업 계획을 수립하고, 단일 문서로 맥락을 유지하는 스킬입니다.

## 트리거

- "플래닝", "계획 세워줘", "어떻게 진행할까"
- "로드맵 만들어줘", "설계해줘"
- "리팩토링 계획"
- 기능 추가, 화면 추가, 여러 파일에 걸친 변경 요청 시

## 문서 관리

plan 문서는 레포 루트의 `.docs/plans/`에 저장한다. 단일 앱이므로 프로젝트별 분리는 없다.

```
.docs/plans/
├── {작업명}.md            ← 진행 중
├── {작업명}.backlog.md    ← 짝 피드백 원장
└── completed/             ← /ship 완료 시 이동
```

- **진행 중**: `.docs/plans/{작업명}.md`
- **완료**: `/ship` 후 `.docs/plans/completed/`로 이동
- **작업명**: kebab-case (예: `book-search`, `wishlist-persist`)

### Backlog 짝 파일 = 피드백 단일 원장 (의무)

각 plan은 **짝 backlog 파일**을 가진다. backlog는 plan 진행 중 받은 **모든 사용자 피드백의 단일 진입점(ledger)**이다. 작업 1건 = `plan.md` + `backlog.md` **2파일로 고정**, ship까지 함께 누적된다.

| 파일 | 위치 | 역할 |
|---|---|---|
| `{작업명}.md` | `.docs/plans/` | 체크리스트·결정·검증 (작업 순서) |
| `{작업명}.backlog.md` | `.docs/plans/` (plan 옆) | **모든 피드백 원장** + 라우팅 + 컨벤션 개선 후보 식별 |

#### 🚫 절대 규칙 — 진행 중 plan이 있으면 새 plan 파일을 만들지 않는다

진행 중인 plan이 있는 상태에서 사용자 피드백을 받으면, **새 `.docs/plans/*.md`를 생성하는 것은 기본 금지**다. 모든 피드백은 먼저 짝 backlog 원장에 한 줄로 기록한 뒤 아래 라우팅 규칙으로 처리한다. (과거 안티패턴: 피드백마다 파일이 파편화됨.)

#### 피드백 라우팅 규칙 (모든 피드백 공통)

```
사용자 피드백 수신
  → ① backlog "피드백 원장" 표에 무조건 한 줄 기록 (받은 내용 그대로)
  → ② 주제 판정
       ├─ on-topic  → 현재 plan에 Step/Phase 추가·수정 → 원장 상태 ✅
       ├─ off-topic → backlog "⏳ 별도 plan 후보" 절에 기록만. 새 파일 생성 ❌ → 상태 ⏳
       │              (ship 단계에서 분리 여부를 사용자에게 질의)
       └─ 사용자가 "지금 따로 빼줘" 명시 → 즉시 새 plan+backlog 쌍 생성
                                          + 현재 backlog에 역링크 → 상태 🔀
  → ③ 3회+ 반복 또는 구조적 패턴이면 추가로 💡 컨벤션 개선 후보 분류
```

**on/off-topic 판정 기준**: 현재 plan의 "목표" 한 문장에 부합하면 on-topic. 별개 기능·다른 레이어·독립 shippable이면 off-topic.

#### backlog 문서 구조 (표준 — Phase 2.5 빈 템플릿과 동일)

```markdown
# {작업명} backlog

> plan(`{작업명}.md`)의 피드백 원장(ledger). 모든 피드백을 여기 먼저 기록 후 라우팅. plan과 짝으로 ship까지 누적 — 길 잃지 않기 위한 SOT.

## 상태 범례
- ✅ 반영 완료 (현재 plan에 박힘)
- 🔀 분리됨 (→ 별도 plan, 사용자 명시 시 즉시)
- ⏳ 별도 plan 후보 (off-topic — ship 시 분리 검토)
- 💡 컨벤션 개선 후보 (규칙 승격 — /review에서 반영 판단)

## 피드백 원장
| # | 받은 피드백 | 주제 | 라우팅 | 상태 |
|---|---|---|---|---|
| F-1 | … | on/off | 현재 plan Phase N / 별도 후보 / → 분리파일 | ✅/⏳/🔀 |

## ⏳ 별도 plan 후보 (off-topic — ship 시 분리 검토)
| # | 피드백 | 이유 | 분리 시점 |
|---|---|---|---|

## 💡 컨벤션 개선 후보
- 룰/컨벤션 신설·확장 가치가 있는 항목
```

**운영 규칙**:

- plan 작성 시 backlog 파일도 같이 신설 (빈 템플릿) — 소규모라도 예외 없음
- 피드백 받을 때마다 **원장 표에 먼저 한 줄** → 라우팅 → 반영 즉시 상태 마킹
- plan 본문 "발견 사항" 절은 backlog로 단순 링크 (중복 방지)
- 1회성 선호는 원장에만, **3회+ 반복/구조적 패턴**은 `💡 컨벤션 개선 후보`로 분류 → `/review`의 "컨벤션/스킬 개선 제안" 단계에서 반영 판단

## 실행 절차

### Phase 0: 요구사항 명확화 (Clarify — 기본 실행)

작업 시작 전 결정 트리의 각 브랜치를 하나씩 탐색해 진짜 목표를 파악한다. **Karpathy 원칙 1 (Think Before Coding) 강제 적용**. 가정을 명시하고 트레이드오프를 제시한다. 모호한 부분이 남아 있으면 질문한다. 상세: `.claude/rules/karpathy-principles.md`.

> **원칙**: Phase 0를 건너뛰는 것이 예외이며, 스킵 시 사용자에게 "스킵 사유: {조건}"을 한 줄 보고한다. 불확실하면 항상 실행한다.

**질문 방식:**

- 질문 **하나씩** — 한번에 여러 개 쏘지 않음
- 각 질문에 **추천 답변 포함** ("저라면 X 선택하겠습니다, 이유: ...")
- 코드베이스에서 답할 수 있는 것은 탐색으로 확인 후 제시
- 결정 트리의 모든 브랜치가 resolve될 때까지 계속

```
사용자: "검색 기능 추가해줘"

Claude: 무엇을 기준으로 검색하시겠어요?
→ 추천: 카카오 도서 API의 제목(title) 검색 (target=title 기본, 현재 검색 UI 없음 확인)

사용자: 제목이요

Claude: 실시간 검색(타이핑마다)과 버튼/엔터 후 검색 중 어느 쪽인가요?
→ 추천: 엔터/버튼 후 검색 (카카오 API 호출량 절약, 페이지네이션과 일관)

...결정 트리 완전 탐색 후 plan 문서 작성
```

#### 스킵 조건 (전부 만족 시에만)

아래 4개 조건을 **모두 만족**할 때만 Phase 0 스킵 허용:

1. **단일 동작 대상**: 변경 대상이 명확한 단일 파일/함수 (예: "useBookSearch에서 handleSubmit 분리해줘")
2. **모호 어휘 부재**: 요청에 `개선`/`최적화`/`정리`/`더 좋게`/`추가` 같은 모호 동사가 없거나, 있어도 대상이 분명
3. **트레이드오프 부재**: 구현 방향이 1가지로 자명 — 두 가지 이상 방법이 떠오르면 스킵 금지
4. **사용자 결정 부재**: API 파라미터/디자인/UX 결정처럼 사용자만 답할 수 있는 항목이 없음

조건 1개라도 어긋나면 Phase 0 실행. 스킵 시 plan 문서 "결정 사항"에 `Phase 0 스킵: {조건 충족 사유}` 1줄 기록.

**규칙:**

- 질문 도중 확정된 결정은 plan 문서 "결정 사항"에 즉시 기록
- 사용자 답변이 추천과 다르면 추천을 폐기하고 답변을 채택 (질문은 진실 발견 도구지 설득 도구가 아님)

- 구체적인 코드 예시는 기존 코드에서 추출

### Phase 2.5: backlog 짝 파일 즉시 신설 (의무)

> plan.md Write 직후, **Phase 3 진입 전** 같은 디렉토리에 `{작업명}.backlog.md`를 빈 템플릿으로 신설한다. 미신설 시 Phase 3 진입 금지.

이 단계는 위 "Backlog 짝 파일 = 피드백 단일 원장" 정책의 실행 게이트. 짝 backlog 누락 재발 방지를 위한 강제 단계.

**빈 템플릿** (`{작업명}.backlog.md`) — 위 "backlog 문서 구조"와 동일:

```markdown
# {작업명} backlog

> plan(`{작업명}.md`)의 피드백 원장(ledger). 모든 피드백을 여기 먼저 기록 후 라우팅. plan과 짝으로 ship까지 누적 — 길 잃지 않기 위한 SOT.

## 상태 범례

- ✅ 반영 완료 (현재 plan에 박힘)
- 🔀 분리됨 (→ 별도 plan, 사용자 명시 시 즉시)
- ⏳ 별도 plan 후보 (off-topic — ship 시 분리 검토)
- 💡 컨벤션 개선 후보 (규칙 승격 — /review에서 반영 판단)

## 피드백 원장

| # | 받은 피드백 | 주제 | 라우팅 | 상태 |
|---|---|---|---|---|
| F-1 | (첫 피드백) | on/off | 현재 plan / 별도 후보 / 분리 | ⏳ |

## ⏳ 별도 plan 후보 (off-topic — ship 시 분리 검토)

| # | 피드백 | 이유 | 분리 시점 |
|---|---|---|---|
| (없으면 절 유지 — off-topic 발견 시 추가) | | | |

## 💡 컨벤션 개선 후보

(빈 상태)
```

**의무 사항**:

- plan.md와 같은 디렉토리에 `{작업명}.backlog.md` 즉시 신설 (소규모 변경이라도 예외 없음)
- `## ⏳ 별도 plan 후보` 절은 항상 포함 (라우팅 규칙)
- 피드백 받을 때마다 **원장 표에 먼저 한 줄** 누적 — 라우팅 후 상태 마킹

**자동 감사**:

```bash
# plan 신설 후 짝 backlog 누락 검사
for plan in .docs/plans/*.md; do
  [[ "$plan" == *.backlog.md ]] && continue
  [[ "$plan" == *_Template.md ]] && continue
  [[ "$plan" == *README.md ]] && continue
  backlog="${plan%.md}.backlog.md"
  [ -f "$backlog" ] || echo "MISSING: $backlog"
done
```

### Phase 3: Pre-flight Plan Review (3 렌즈 사전 검증)

plan.md 생성 직후, 실행 전에 3가지 관점으로 plan 문서를 교차 검증한다.

> **목적**: 구현 후 발견 시 재작업 비용이 큰 문제를 Plan 단계에서 잡는 것. 코드 탐색 없이 plan 문서만 읽어 빠르게 수행.
> 뒤의 `/review` `/review-ui`가 실제 구현을 검증하는 실제 게이트다. Pre-flight는 **"방향이 맞는가"만** 체크.

각 렌즈는 Claude가 해당 역할로 관점을 전환하여 수행한다 (코드 탐색 없이 plan 문서만 읽음).

#### 🎯 기획자 렌즈 — "맞는 걸 만드는가?"

- [ ] 요구사항에 모호한 동사가 없는가 ("개선", "최적화", "더 좋게")
- [ ] Must 항목이 전체의 60% 이하인가 (스코프 인플레이션 방지)
- [ ] 사용자가 결정해야 할 사항이 "결정 사항" 섹션에 명시됐는가

#### 🖥️ 프론트엔드 렌즈 — "구현 가정이 타당한가?"

- [ ] 카카오 도서 API의 응답 shape/파라미터가 Plan에 명시됐는가 (임의 가정 금지)
- [ ] 라우팅/폴더 구조 계획이 규칙과 맞는가 (App Router, 페이지 3파일 원칙)
- [ ] 상태관리가 React Query + nuqs 패턴과 맞는가 (Query hook 구조분해 금지, Controller 통일)

#### 🎨 디자이너 렌즈 — "기존 시스템과 맞는가?"

- [ ] Plan에 언급된 UI 패턴이 기존 `components/`에 존재하는가 (중복 구현 방지)
- [ ] 디자인 토큰(Tailwind `@theme`)과 충돌하는 하드코딩 값이 없는가
- [ ] 반응형/모바일 흐름이 필요한 경우 plan Step에 반영됐는가

#### 결과 보고 포맷

```
## Pre-flight 결과

🔴 BLOCK (N건)
- [렌즈] 이슈 — 영향 설명

🟡 RISK (N건)
- [렌즈] 이슈 — 영향 설명

→ PROCEED / REVISE [BLOCK N건 해소 후 재검증]
```

#### 게이트 규칙

- **BLOCK 0건**: RISK를 plan "발견 사항"에 기록 → 사용자에게 plan 제시 + 승인 대기
- **BLOCK 1건+**: 자동 중단 → 결과 보고 → 사용자 수정 대기 → 수정 완료 시 Pre-flight 재실행
- RISK는 "발견 사항"에 기록 후 진행 (강제 중단 없음)

### Phase 4: 자동 실행 + 점진적 검증

> 사용자가 계획을 승인하면, **별도 지시 없이** 전체 Step을 순차 실행한다.

**실행 모드:**

이 과제는 규모가 작으므로(단독 앱, 페이지 2~3개) **단일 세션 직접 구현이 기본**이다. 별도 에이전트 팀을 구성하지 않는다.

- **직접 구현** (기본): plan의 Step을 순차로 직접 구현·검증
- **reviewer 에이전트** (선택): 구현 후 리뷰의 객관성이 필요할 때만 read-only 리뷰어 에이전트를 1회 spawn해 교차 검증. 그 외에는 `/review`를 직접 실행

**자동 실행 루프:**

```
사용자 승인
  → Step 1 구현 → 검증 → 문서 체크 ✅
  → Step 2 구현 → 검증 → 문서 체크 ✅
  → ...
  → 전체 Step 완료
  → Phase 5 자동 진입 (리뷰 → 커밋까지 일괄)
```

**Step 실행 규칙:**

1. 문서에서 다음 미완료 Step 확인
2. 구현
3. Step별 검증 실행
4. 문서의 체크리스트 즉시 업데이트 (`- [ ]` → `- [x]`)
5. 예상 밖의 결정 → "결정 사항" 섹션에 기록
6. 새 패턴 발견 → "컨벤션 변경 필요" 섹션에 기록
7. 다음 Step

**실행 중 문서 동기화:**

Step 실행 도중 계획과 다른 상황이 발생하면, **코드뿐 아니라 plan 문서도 즉시 업데이트**한다.

| 상황                       | 문서 반영                      | 섹션                            |
| -------------------------- | ------------------------------ | ------------------------------- |
| 디버깅 중 반복 이슈 발견   | 원인+해결책 기록               | `발견 사항`                     |
| 계획에 없던 기능 추가 필요 | 새 Step 삽입 (번호 조정)       | `체크리스트` + `수정 파일 목록` |
| 기존 Step이 불필요해짐     | ~~취소선~~ + 사유 기록         | `체크리스트` + `결정 사항`      |
| Step 범위가 예상보다 큼    | Step 분할 (1.1 → 1.1a, 1.1b)   | `체크리스트`                    |
| 기존 요구사항 변경         | 요구사항 섹션 수정 + 변경 이력 | `요구사항` + `결정 사항`        |
| 새로운 위험 발견           | 위험 항목 추가                 | `실패 위험 (Pre-mortem)`        |
| 컨벤션 위반/개선 패턴 발견 | 패턴 기록                      | `컨벤션 변경 필요`              |

**핵심 규칙:**

- 문서는 **항상 현재 상태의 진실**이어야 한다 (과거 계획이 아님)
- Step 추가/삭제 시 사용자에게 한 줄 알림 ("Step 2.3 추가: xxx 필요")
- 대화가 끊겨도 문서만 읽으면 현재 상태를 완전히 파악 가능해야 함

**실패 시:**

- Step 실패 → 원인 분석 → "발견 사항"에 기록 → 수정 후 재시도 (최대 3회)
- 3회 실패 → **중단하고 사용자에게 보고**

**사용자에게 질문이 필요한 경우 (자동 실행 일시 중단):**

- 구현 방향이 2가지 이상이고 트레이드오프가 명확할 때
- 기존 요구사항과 충돌하는 상황 발견 시
- 계획에 없던 대규모 변경이 필요할 때
- Step 실패 3회 시

→ 사용자에게 상황 설명 + 선택지 제시 → 답변 후 자동 실행 재개
→ 답변 내용은 문서의 "결정 사항"에 기록

**위 경우 외에는 끝까지 자동 진행한다.**

### Phase 5: 자동 검증 + 완료 (Verify — Goal-Driven Execution)

모든 Step 완료 후, 변경된 파일을 기반으로 **필요한 검증을 자동 판단**하여 실행한다. 각 검증은 **"[단계] → 검증: [확인 방법] → 결과: [확인 결과]"** 포맷으로 보고한다 (Karpathy 원칙 4).

**1. 기본 (항상 실행)**

- 체크리스트 최종 확인 — 모든 항목 ✅
- Pre-mortem 항목 재확인
- `/review` 실행 — 컨벤션 체크 + 타입체크 + 린트

**2. 보안 검증**

별도 `/security` 자동 실행 없음 — `/review`의 Security 차원(가중 0.30)이 같은 diff에 보안 SOT 체크리스트(`.claude/skills/security/SKILL.md`)를 적용하므로 이중 패스가 된다. 심층·범위 지정 보안 리뷰가 필요하면 사용자가 `/security [범위]`를 단독 호출.

**3. UI 검증 (자동 판단)**

변경 파일에 아래 패턴이 포함되면 `/review-ui` 자동 실행:

- `app/**/page.tsx`, `app/**/page.style.ts` (페이지)
- `components/**` (UI 컴포넌트)

**4. 결과 보고 — Goal-Driven 포맷**

각 단계마다 `→ 검증:` + `→ 결과:`를 명시한다. "동작합니다" 추상 보고 금지.

```
## Phase 5 검증 결과

### Step 검증 매트릭스
| Step                       | 검증 방법                       | 결과                              |
| -------------------------- | ------------------------------- | --------------------------------- |
| Step 1.1 (도서 검색 API)   | integration test (MSW + shape)  | ✅ PASS (5 tests)                 |
| Step 1.2 (도서 카드 UI)    | review-ui DOM 정렬 + 4px grid   | ✅ Accessibility 94 / Performance 90 |

### /review (점수)
| 차원            | 점수 | 등급 | → 검증                                |
| --------------- | ---- | ---- | ------------------------------------- |
| Correctness     | 100  | A    | typecheck 통과                        |
| Security        | 100  | A    | OWASP grep + 입력 검증                |
| Maintainability | 97   | A    | Karpathy Surgical Changes 위반 0건    |
| Consistency     | 90   | A    | conventions 패턴 매칭                 |
| Performance     | 100  | A    | 참조 안정성 grep 0건                  |
| **종합**        | 98   | A    | Quality Gate PASS                     |

### 기타 검증
- /review-ui → 검증: DOM 정렬 + a11y + Lighthouse — 결과: A11y 94, Perf 90, Consistency 90

### Phase 5 → /ship 전환
→ 검증: Blocking 0건 + 종합 ≥ 70 — 결과: PASS → /ship 자동 실행
```

실행한 검증과 **실행하지 않은 검증(사유 포함)**을 모두 사용자에게 명시한다. 예: `/review-ui 스킵 → 검증 대상 파일(UI 컴포넌트/페이지) 없음`.

**5. Quality Gate 통과 시 자동 `/ship`**

- Blocking 0건 + 종합 70점 이상 → `/ship` 자동 실행 (린트 → 타입체크 → 브라우저 검증 → 커밋 → plan 문서 completed/ 이동)
- Quality Gate 미통과 → 이슈 수정 후 재검증 (최대 2회), 실패 시 사용자에게 보고하고 중단

2. 해당 문서 읽기

## 핵심 원칙

### 점진적 계획

- 한 번에 모든 것을 계획하지 않음
- Phase별로 검증 후 다음 Phase 진행
- 각 Step은 독립적으로 실행/롤백 가능해야 함

### 토큰 효율

- 소규모: 목표 + 체크리스트 + 수정 파일만 (3 Whys, Pre-mortem 스킵)
- 중규모: 전체 간소화
- 대규모: 전체 상세
